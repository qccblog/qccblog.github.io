    bit 的基本概念

    qiucong 2021/5/20 bit安装配置

    Bit (opens new window) 是一个用于从独立组件组合出现代应用程序的开源工具链。在 Bit 中，构建的是组件而不是应用程序。在工作区中只有组件，应用程序只是另一个组件。一个组件可以是许多其他组件的组合，最终组合成整个应用程序。在 Bit 中，从来没有项目，也没有应用程序，因为拥有应用程序违背了组件驱动架构的本质。

    在 Bit 的思想中没有应用程序，有的只是大大小小的组件然后就像搭积木一样搭建，专注于组件。不管这个想法是否符合你的思维，把 Bit 当成一个工具在不同项目中共享和同步组件是一个不错的方案。

    本文的内容主要是一些在看文档过程中记的一些杂七杂八的内容，很多东西在官方文档中描述的更为细致和准确，希望能让大家对 bit 有个大概的了解就好了。如果只是想要简单使用 Bit，可以查看 bit 的简单使用

    # 安装

    v15-Harmony 版本相对 V14 是一个改动比较大的版本，两者在使用方式上也存在一些差异，官方目前(2021-05)是建议使用 React 和 Nodejs 的使用 v15 (opens new window)，如果是用 Vue 和 Angular 建议还是使用 v14 (opens new window)。

    BVM (opens new window) 是新推出来的一个 Bit 版本管理工具，可以方便的安装和管理多个版本的 Bit。

    # 安装bvm
npm i -g @teambit/bvm

# 安装最新版Bit
bvm install

# 升级Bit
bvm upgrade

    # 初始化 Workspace

    Bit Workspace 可以用来方便的创作和管理多个独立的组件。Bit 与 Git 一起使用，因此可以将 Bit 添加到新的或已有的 git 仓库中。

    # 目录下执行
bit init --harmony

    在初始化 workspace 的时候会自动创建一些文件：

    • workspace.jsonc - workspace 配置文件，用于为工作区及其所有组件设置规则和策略。
    • .bitmap - 工作空间中被跟踪的组件与其在文件系统上的物理位置之间的自动生成的映射。工作区的文件结构完全取决于您。
    • .git/bit（隐藏目录）-您的本地范围。workspace 的组件发行版本的存储位置。（ps：实测先有 git 则是.git/bit，现有 bit 则是.bit 和.git）

    目前还提供一种实验性质的快速初始化方式，可以用来快速创建一个 React 环境的 workspace。

    # 创建一个新的bit项目
bit new react-workspace <my-workspace-name>

# 安装依赖
cd <my-workspace-name>
bit install

    bit create 以 react-component 为模板在 ui 这个命名空间（namespace）中华创建了一个 button 组件，同时在 node_modules 和 bitmap 文件中添加了创建的组件。组件创建在 my-scope/ui/button，my-scope 是 defaultScope，默认的目录就是{scope}/{name}。

    bit 默认使用 pnpm 作为依赖解析工具，可以选择则 yarn 或者 pnpm。使用 bit 之后被建议使用 bit 作为依赖管理工具，即后续要安装什么 npm 包使用 bit 进行安装，如果是在某个工程第一次安装依赖也可以直接使用 bit install，bit 会查找 package.json 和 workspace.jsonc 中要安装的依赖，如果有相同的依赖则以 workspace.jsonc 中的为准。ps：使用 bit 安装的 npm 包是来自 Bit.dev 的仓库，所以要用 Bit 的登录认证。

    也可以不适用 bit 作为包管理工具，直接使用 npm 或者 yarn 之类的，需要注意两点：

    • 在 Bit 生成组件的时候使用 bit link 作为 post-install 脚本来 生成组件模块链接 (opens new window)
    • Bit 支持不同组件使用相同依赖的不同版本，但是一般的包管理工具不支持这个功能

    启动 dev 服务后，默认情况下访问 localhost: 3000 会打开组件的 UI 展示界面，上面有关于组件的各自信息。

    # 使用组件

    不允许通过相对路径来使用组件，只能通过组件的包名来使用组件，这样可以保证后续移动组件位置之类的操作不会影响到使用的地方。bit 在构造组件的过程中会自动添加到 node_modules，所以可以直接以依赖包名的方式引入。

    import { Button } from '@my-scope/ui.button'
    1

    # 创建远端 Scope

    Scope 就是存储 Bit 组件发行版本的地方，默认初始化一个 workspace 的时候就在本地创建了一个默认的 my-scope。但是如果只是本地的 scope，不同项目之间就不能方便的共用组件，所以需要一个远端的 Scope，官方提供的 Bit.dev (opens new window) 或者自己搭建的 Bit 服务都是可以作为远端 Scope 的。

    # bit.dev

    # 注册账号

    先注册一个 bit 的账号，然后再 创建 (opens new window) 一个 collection（差不多就是 scope）。

    # 本地工程登录

    在本地工程目录下执行 bit login 会打开一个浏览器标签，如果已经登录会显示登录成功然后几秒后跳转到 https://bit.dev/ (opens new window)，没登录就要先进行登录，登录成功会提示成功再跳转。这个时候本地工程的命令行也会提示登录成功，显示登录的用户名。

    # 更新本地的默认 Scope

    更新本地的默认 Scope 为你刚刚创建的 collection 的名字，修改 workspace.jsonc 中 defaultScope 的值，把 my-scope 改成刚刚创建的 username/organization name 和 'demo-scope'作为 scope。

    {
  "teambit.workspace/workspace": {
    "defaultScope": "your-username.demo-scope"
  }
}

    # 发布组件

    如果远端 Scope 配置完成之后就可以把本地的组件发布到远端，然后其他项目就可以安装使用了。

    # 给全部改动的组件打tag
bit tag --all --message "first version"

# 发布组件
bit export

    发布成功之后就可以在远端看到发布的组件了。

    # 导入(import)组件 (opens new window)

    导入就是从远程 Scope 完整导入某个组件到本地 workspace 中，本地可以修改、维护甚至发布新的版本到远程 Scope 上。

    通过组件的 ID 来进行导入，ID 由 scope-owner.scope-name/namespace/component-name 组成

    bit import <scope>/<component>

# 从叫teaching的scope中namespace为ui/elements中导入组件dots-loader
bit import teambit.teaching/ui/elements/dots-loader

# 从叫teaching的scope中namespace为ui下的组件
bit import teambit.teaching/ui/*

# 从叫teaching的scope导入全部组件
bit import teambit.teaching/*

# 获取已经导入过组件的最新版
bit import

    # 测试组件 (opens new window)

    测试用例需要添加在组件的目录下，测试的文件名要匹配组件环境中的定义，比如 react 的测试文件名要符合 *.spec.[ts|tsx|js|jsx] 和 *.test.[ts|tsx|js|jsx]。强烈推荐使用组件的 compositions 实例作为用例的实例。在一个 workspace 中可以给不同的组件在环境中配置不同的执行测试的工具。

    # 测试某个组件
bit test <component-id>

# 测试全部组件
bit test

    # 不同 Scope 之间依赖

    如果有组件依赖不同的 Scope，需要建立内部依赖。假如 A 中有一个组件依赖 B 中的组件，需要把 B 作为 A 的 remot。

    bit remote add file:///your-scope-a-directory
